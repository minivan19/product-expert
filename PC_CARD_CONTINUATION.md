## 二、已确认的标准规范（必须严格遵守）

### 2.1 ID编号规则
- **PC卡ID = 对应SC_ID**（例：SC_08 → PC_08，SC_32 → PC_32）
- 客户卡在标准卡ID后加客户前缀，如：`PC_32_导远`

### 2.2 卡片文件命名
- 路径：`/Users/limingheng/.openclaw/workspace/skills/product-expert/framework/product_solution_cards/PC_{ID}.json`
- 示例：`PC_32.json`

### 2.3 顶层JSON结构（8个必填字段）
```json
{
  "id": "PC_XX",
  "名称": "方案名称",
  "版本": "v1",
  "状态": "草稿",
  "来源": {
    "类型": "手工创建 | 蓝图提取 | 场景三生成",
    "来源文件": "文件名",
    "提取时间": "YYYY-MM-DD"
  },
  "对应业务卡": ["SC_XX"],
  "来源SC方案结构": {
    "SC_ID": "SC_XX",
    "描述": "SC方案一句话描述"
  },
  "业务流程说明": "一句话说明SRM侧做什么（外部系统不计入SRM范围）",
  "适用条件": {
    "行业": ["汽车/汽配", "..."],
    "物料属性": ["关键品", "..."],
    "权力结构": ["强势买方", "..."],
    "协同成熟度": ["中", "高"]
  },
  "业务流程": [ /* 见2.4 */ ],
  "系统流程方案": [ /* 见2.5 */ ],
  "产品功能组合": [ /* 见2.6 */ ],
  "实施步骤": [ /* 见2.7 */ ],
  "二开介入点": [ /* 见2.8 */ ],
  "关键配置项": [ /* 见2.9 */ ],
  "成功客户": [],
  "关联方案卡": ["PC_XX", "PC_XX"],
  "更新时间": "YYYY-MM-DD",
  "维护人": ""
}
```

### 2.4 业务流程（每步必须含：步骤/业务动作/责任方/说明）
- 步骤序号从1开始
- **重要原则：外部系统（如S&OP、MRP、WMSOMS、ERP等）不计入SRM业务流程**，只做数据对接
- 外部系统的协同动作写入"说明"中，不作为独立步骤

### 2.5 系统流程方案（每个业务步骤对应一个系统动作）
必填字段：`对应业务步骤`、`系统动作`、`触发模块`、`输出物`、`后续触发`、`二开`、`二开说明`
- 二开说明仅在 `二开: true` 时填写

### 2.6 产品功能组合（字段名固定为"产品功能组合"）
必填字段：`功能`、`套件`、`角色`、`配置项`、`二开介入`
- ⚠️ 字段名用"功能"，不用"模块"

### 2.7 实施步骤（每步必填：顺序/步骤/产品模块/前置依赖/二开）
- 实施步骤中的"产品模块"字段名称不变（实施层面的通用描述）
- 顺序字段为整数

### 2.8 二开介入点
- 如无二开，写 `[]`（空数组），不写 `null`
- 如有二开，每项含：`环节`、`描述`、`难度`、`预估人天`

### 2.9 关键配置项
- 格式：数组，每项为字符串
- 描述"数据来源"时，必须写清楚SRM侧负责什么，外部系统负责什么

### 2.10 索引更新
入库后必须同步更新 `product_card_index.json`：
- 新增 `已有卡片` 数组项
- 更新 `汇总`（总卡数+1）
